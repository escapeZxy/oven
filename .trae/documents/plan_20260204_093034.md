# 重构计划：引入 ScheduleItem 支持线性混合排课

为了支持“循环”与“单次训练日”的自由混合编排，我们将对核心数据模型和逻辑层进行重构。此变更涉及数据结构定义的升级、业务逻辑的适配以及对现有数据的兼容性处理。

## 1. 核心模型重构 (Data Model)
**目标**：引入 `ScheduleItem` 联合类型，使 `WorkoutPlan` 能够容纳混合结构。

- **修改文件**：`packages/core/src/models/plan.model.ts`
- **变更详情**：
  1.  定义 `TrainingDay` 接口，增加 `type: 'day'` 标识符。
  2.  新增 `TrainingCycle` 接口，包含 `type: 'cycle'`、`repeats`（重复次数）和 `items`（嵌套调度项）。
  3.  定义 `ScheduleItem = TrainingDay | TrainingCycle`。
  4.  更新 `WorkoutPlan` 接口，新增 `schedule: ScheduleItem[]` 字段。
  5.  保留 `trainingDays` 字段并标记为 `@deprecated`，用于向后兼容。

## 2. 业务逻辑适配 (Service Layer)
**目标**：确保业务逻辑能够理解并处理新的嵌套结构，将其转化为线性的可执行序列。

- **修改文件**：`packages/core/src/services/workout.service.ts`
- **变更详情**：
  1.  实现 `flattenPlan(plan: WorkoutPlan): TrainingDay[]` 私有方法。
      - 递归遍历 `plan.schedule`。
      - 如果遇到 `Cycle`，根据 `repeats` 次数重复展开其内部的 `items`。
      - 如果遇到 `Day`，直接加入列表。
      - **兼容处理**：如果 `schedule` 为空，回退使用旧的 `trainingDays`。
  2.  更新 `logWorkout` 和 `skipDay` 方法。
      - 在计算 `isLastDay` 和获取 `trainingDayId` 时，先调用 `flattenPlan` 获取完整的线性列表，再基于 `currentDayIndex` 进行操作。

## 3. 前端逻辑更新 (Frontend Access)
**目标**：确保 UI 层能够正确获取今日训练内容，而不感知底层的复杂结构。

- **修改文件**：`src/app/features/workout-log/data-access/workout-log.service.ts`
- **变更详情**：
  1.  更新 `getTodaysWorkout` 方法。
      - 使用 `WorkoutService` 的扁平化逻辑（或在前端复制一份简单的扁平化逻辑）来根据 `currentDayIndex` 定位正确的 `TrainingDay`。
      - *注：为了保持一致性，建议在 `WorkoutPlan` 模型中直接增加一个辅助方法或在 Service 中暴露 `getFlattenedDays(plan)`。*

## 4. 数据迁移与播种 (Migration & Seeding)
**目标**：更新 Mock 数据以展示新结构的威力。

- **修改文件**：`packages/core/src/infrastructure/seeder.ts`
- **变更详情**：
  1.  更新 `MOCK_WORKOUT_PLANS`，将旧数据迁移到 `schedule` 字段。
  2.  创建一个新的示例计划（例如“混合编排示例”），包含 Cycle 和单独 Day 的混合结构，用于测试验证。

## 5. UI 适配 (UI Layer) - *本次暂不深入*
本次重构主要集中在**核心逻辑与数据层**。
- `plan-create` 页面目前仍使用旧的 `trainingDays` 数组。为了不阻塞重构，我们将保持其现状（它生成的计划会被视为只有 `Day` 的简单计划），后续单独安排任务升级 UI 以支持 Cycle 的创建。

## 执行步骤
1.  **Model**: 更新 `plan.model.ts` 定义。
2.  **Logic**: 在 `WorkoutService` 中实现 Flatten 逻辑并替换旧的索引访问代码。
3.  **DataAccess**: 更新 `WorkoutLogService` 适配新逻辑。
4.  **Data**: 更新 Seeder 数据。
5.  **Verify**: 运行测试确保旧功能未破坏，新数据结构可被正确解析。
