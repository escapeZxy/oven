# 历史记录功能增强设计方案

你提出的需求非常合理。为了支持“继续”、“重新开始”和“删除”这三个逻辑，并且清晰区分它们的行为，我们需要对数据状态和业务逻辑进行精细化设计。

## 1. 核心概念澄清

*   **继续 (Resume)**: 针对状态为 `interrupted` 的计划。行为是：把它的状态改回 `active`，并把当前正在运行的其他计划（如果有）改为 `interrupted`。进度（`currentDayIndex`）保持不变。
*   **重新开始 (Restart)**: 针对任何历史计划。行为是：**不修改**这条历史记录，而是基于同一个 `workoutPlanId` 创建一条**全新的** `UserWorkoutPlan` 记录，状态为 `active`，进度归零。
*   **删除 (Delete)**: 针对历史记录。行为是：**软删除**或标记为不可见。考虑到数据价值，建议引入一个 `deleted` 状态或字段，而不是物理删除。

## 2. 数据结构设计 (Data Structure)

我们需要在 `UserWorkoutPlan` 模型中引入一个“归档/删除”的标记，以便在历史记录中隐藏它，但保留数据用于统计。

- **修改文件**: `packages/core/src/models/log.model.ts`
- **变更**:
  ```typescript
  export interface UserWorkoutPlan {
    // ... existing fields
    status: 'active' | 'completed' | 'interrupted';
    
    /**
     * If true, this plan is hidden from the history list (soft deleted).
     * It won't affect active status, but 'active' plans should not be archived.
     */
    isArchived?: boolean; 
  }
  ```
  *(注：使用 `isArchived` 比 `deleted` 语义更温和，适合这种场景)*

## 3. 业务逻辑实现 (Service Layer)

我们需要在 `WorkoutService` 和 `WorkoutLogService` 中实现或增强以下方法：

1.  **Resume (继续)**: 
    - 逻辑：`active` -> `interrupted`; `target(interrupted)` -> `active`。
    - *已在上一轮实现，需确认逻辑是否完美。*
2.  **Restart (重新开始)**:
    - 逻辑：`active` -> `interrupted`; `create new(active)`。
    - *已通过 `startPlan` 实现，因为 `startPlan` 本质上就是基于 PlanID 创建新实例。这里需要确认前端调用的是 `startExistingPlan(planId)` 而不是操作 userPlan。*
3.  **Delete (删除)**:
    - **新增方法**: `archivePlan(userPlanId: string)`
    - 逻辑：设置 `isArchived = true`。前端在获取历史列表时过滤掉这些记录。

## 4. UI 交互设计 (Front-end)

在 `LogListComponent` 的历史记录卡片中，操作按钮将根据状态动态变化：

- **状态: Interrupted (未完成)**
  - [主按钮] **继续**: 调用 `resumePlan(userPlanId)`
  - [次按钮] **删除**: 调用 `archivePlan(userPlanId)`
  - [次按钮] **重新开始**: 调用 `startExistingPlan(workoutPlanId)` (可选，用户可能想放弃进度重来)

- **状态: Completed (已完成)**
  - [主按钮] **重新开始**: 调用 `startExistingPlan(workoutPlanId)`
  - [次按钮] **删除**: 调用 `archivePlan(userPlanId)`
