# 进度归档

## 文档目的

这份文档只记录一件事：

`现在已经做到哪了`

它不是 roadmap。
它记录的是当前系统的真实完成度、已落地能力和还存在的边界。

## 当前阶段结论

项目已经从“前后端开始接通”推进到了“主流程核心链路基本收口”的阶段。

当前最准确的状态是：

- 前端主流程已经完整可用
- 后端已经具备正式数据层和正式业务接口
- 真实认证已经接入主路径
- 训练计划 definition、用户计划、训练日志已经正式持久化
- 首页仪表盘和训练会话已经以后端为正式事实源
- 草稿与正式记录的边界已经明确
- 第一批服务端统计已经落地
- 计划完成率统计已经落地
- 动作趋势统计已经落地
- 多设备同步第一步已经落地
- 训练提交幂等保护已经落地
- 基础冲突恢复已经落地
- 冲突分类语义已经落地
- 最终交付目标已经明确为 `Web + iOS + Android`
- 但正式发布基础、PWA 和原生壳仍未开始

一句话说清：

`本地优先训练应用 -> 已完成正式数据收口 -> 已明确跨端发布目标 -> 正在进入产品化阶段`

## 已完成事项

## 1. 产品主流程

当前已经打通的核心主线：

- 创建训练计划
- 浏览计划详情
- 开始计划
- 进入当天训练
- 记录组数、次数、重量
- 中断后恢复训练草稿
- 完成训练或跳过训练
- 回到仪表盘查看历史与统计

## 2. 前端能力

当前前端已经具备：

- 登录页 / 注册页
- 路由守卫与游客页守卫
- Token 注入与 `401` 自动清会话
- `workout-log` 主流程页面
- 活跃计划展示
- 历史计划展示与筛选
- 计划库展示
- 周容量、月容量统计展示
- 本地训练草稿恢复
- 动作上次正式训练记录回看
- 训练会话中的“完成训练 / 跳过训练”统一走后端原子提交
- 草稿恢复按 `userPlanId + trainingDayId + dayIndex` 校验
- 正式提交只发送已勾选完成的组
- 训练失败可见错误提示
- 仪表盘正式数据同步状态提示
- 手动“立即同步”入口
- 训练会话失败重试时复用同一提交请求键
- 提交冲突后自动同步最新正式数据并保留草稿
- 训练页按冲突类型分流恢复动作

当前前端还没有具备：

- 生产发布环境收口
- PWA 安装壳
- 原生壳适配

`packages/core` 当前已经承担：

## 当前系统状态

## 当前事实源

当前系统已经不再是“读本地、写后端”的灰色双轨状态。

现在更准确的表述是：

- 本地只保留草稿和必要过程态
- 后端承接正式训练记录和正式训练进度
- 后端承接正式计划 definition / 用户计划的增量同步事实源
- 周容量和月容量已改为后端聚合
- 计划完成率已改为后端聚合
- 动作趋势已改为后端聚合
- 前端只负责把聚合结果适配为仪表盘展示

这不是最终形态，但主流程事实源已经基本清楚。

## 当前产品状态

可以明确说：

- 它已经不是空 UI
- 它已经不是本地 mock 玩具
- 它已经具备正式后端数据链路
- 它已经具备原子提交和边界校验
- 它已经具备第一批服务端统计
- 它已经具备计划完成率统计
- 它已经具备动作趋势统计
- 它已经具备正式数据的最小多设备同步拉取
- 它已经具备训练提交幂等保护
- 它已经具备基础冲突恢复
- 它已经具备冲突分类与差异化恢复
- 它已经明确最终会走 `Web + iOS + Android` 三端统一交付

但也必须明确说：

- 它还没有完整多设备同步
- 它还没有完整统计分析体系
- 它仍然缺少更深层训练分析和同步能力
- 它还没有正式生产发布基础
- 它还没有 PWA
- 它还没有 iOS / Android 原生壳

## 已知边界

## 1. 服务端统计只完成第一批

当前已经完成：

- 周容量
- 月容量
- 计划完成率
- 动作趋势

当前还没有：

- 更深层训练分析
- 更细粒度的统计扩展与复用

## 2. 多设备同步只完成第一步

当前已经有：

- `workout-plans` / `user-plans` 的 `updatedAt` 增量拉取
- 前端按 `id` 的增量合并
- 仪表盘同步状态提示
- 手动同步入口
- 训练提交的请求级幂等保护
- 提交冲突后的自动同步恢复
- 冲突类型化返回语义

当前还没有：

- 复杂冲突合并
- 跨设备草稿策略
- 离线重放

## 3. 跨端发布能力还没开始落地

最终目标虽然已经明确成 `Web + iOS + Android`，但当前还没有：

- 生产环境 API / 部署配置收口
- PWA manifest / service worker
- 移动端 Web 关键交互适配验收
- iOS / Android 原生壳与商店发布基础

## 当前最重要的技术判断

现在最该做的不是“扩更多功能”，而是：

- 先把生产 Web 发布基础和移动端 Web 适配做实
- 再补最小 PWA 和完整同步
- 最后再进入 iOS / Android 商店发布准备
