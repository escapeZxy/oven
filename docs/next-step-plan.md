# 下一步执行计划

## 文档目的

这份文档只回答一个问题：

`接下来应该按什么顺序继续做`

它不是长期 roadmap，而是基于当前状态的短中期执行计划。

## 当前判断

项目当前最关键的问题已经不是“前后端主流程没接上”。

现在的主线已经基本收口，当前最大的剩余问题是：

- 最终目标已经明确为 `Web + iOS + Android`
- 当前还没有生产 Web 发布基础
- 当前还没有移动端 Web / PWA 交付壳
- 正式数据跨设备一致性仍未补完整

所以接下来的工作重点应该围绕这件事展开：

`在正式数据链路、认证和核心服务端统计都已落地的前提下，先把 Web 发布做实，再逐步进入 PWA 和 iOS / Android 交付`

## 执行原则

后续执行继续遵循四条规则：

- 先收紧边界，再扩新能力
- 先解决真实问题，不做抽象表演
- 任何新增改动都不能破坏现有用户可见流程
- 所有新能力都必须建立在单一正式事实源之上

还要补一条发布原则：

- 不为上架单独重写第二套前端

## 阶段 D：草稿与正式记录边界收口

当前状态：已完成

### 已完成结果

- 草稿只保存在前端本地

### 当前结论

这一步做完后，主流程已经从“双轨过渡态”进入“正式事实源明确”的阶段。

## 阶段 F：引入真实认证

当前状态：已完成

### 目标

把 mock user 替换成真实用户体系。

### 为什么排在这里

因为现在认证终于不再只是给本地应用套壳。

正式训练记录、正式训练进度和计划 definition 都已经进入后端正式数据层，这时再做认证才有意义。

### 应完成

- 注册 / 登录
- 用户会话管理
- 基础鉴权
- 用户隔离

### 已完成结果

- mock user 已从前端主路径移除
- 登录 / 注册页面已接入
- 会话恢复和注销已接入
- `user-plans` / `workout-logs` 已改为从认证上下文解析用户

## 阶段 G：服务端统计与分析

当前状态：已完成

### 目标

把现在前端做的基础统计逐步迁移到后端。

### 应完成

- 周容量统计 API
- 月容量统计 API
- 计划完成率统计
- 动作趋势统计

### 已完成结果

- 周容量统计 API 已落地
- 月容量统计 API 已落地
- 计划完成率统计已落地
- 动作趋势统计已落地
- 前端仪表盘已切到服务端聚合结果
- 前端图表组件保持稳定，仅保留展示职责

### 本阶段剩余项

- 更深层训练分析
- 统计接口的进一步扩展与复用

### 前提

必须建立在：

- 真实用户体系稳定
- 正式日志稳定
- 用户计划稳定

之上。

## 阶段 H：多设备同步

当前状态：第一步已完成

### 目标

让不同设备之间共享正式数据。

### 为什么最后做

因为同步会强迫系统面对：

- 幂等保护
- 冲突处理
- 时间戳策略
- 离线重放

这些问题只有在认证和正式数据边界都稳定后才值得处理。

### 已完成结果

- `GET /workout-plans` 已支持 `updatedAfter` / `limit`
- `GET /user-plans` 已支持 `updatedAfter` / `limit`
- `GET /workout-logs` 已支持 `updatedAfter` / `limit`
- 前端已接入正式数据同步游标
- 仪表盘已显示同步状态并提供手动同步入口
- 正式数据已支持按 `id` 的增量合并
- `POST /user-plans/:id/logs` 已支持请求级幂等保护
- 提交冲突后会自动同步最新正式数据并保留本地草稿
- 提交冲突已支持 `code + recoveryAction` 分类语义

### 当前剩余

- 复杂冲突合并
- 跨设备草稿策略
- 离线重放
- 更完整的同步错误恢复策略

## 阶段 I：生产 Web 发布基础

当前状态：下一阶段

### 目标

让产品先以正式 Web 形态可部署、可访问、可维护。

### 应完成

- 生产环境 API 地址收口
- 后端生产环境变量收口
- 数据库部署方案明确
- 前端静态托管方案明确
- 后端服务托管方案明确
- 域名、HTTPS、CORS、健康检查可用

### 为什么排在现在

因为没有正式 Web 发布基础，后续 PWA 和原生壳都没有落脚点。

## 阶段 J：移动端 Web 与最小 PWA

当前状态：未开始

### 目标

让同一套 Angular 前端在手机和平板上可用，并具备最小安装能力。

### 应完成

- 关键页面移动端适配
- 安全区和底部操作栏适配
- `manifest.webmanifest`
- 图标和 `theme-color`
- service worker 或等价静态壳缓存

### 当前边界

- 这一阶段只做可访问、可安装
- 不在这一阶段引入完整离线提交能力
- 不在这一阶段改变正式事实源

## 阶段 K：iOS / Android 原生壳与商店发布

当前状态：未开始

### 目标

在不重写业务前端的前提下，把产品送进 iOS / Android 商店。

### 应完成

- 选定原生壳路线，例如 Capacitor
- 生成 iOS / Android 工程
- 图标、启动图、包名、签名、提审配置
- 真机验证
- TestFlight / Play 内测流程跑通

## 明确暂时不做的事

下面这些事情现在不要碰：

- 饮食系统
- 体重系统
- 社交系统
- 推荐系统
- 微服务拆分
- 通用同步引擎
- 复杂 BI 报表平台

原因只有一个：

这些都不解决当前主线问题。

## 最近一轮建议执行顺序

如果只看接下来这一轮开发，建议按下面顺序：

1. 先完成生产 Web 发布基础
2. 再补移动端 Web 适配和最小 PWA
3. 再补复杂冲突合并和更完整的同步错误恢复
4. 再明确跨设备草稿仍然不进正式同步
5. 最后再进入 iOS / Android 原生壳与商店发布
6. 再扩更深层分析

## 成功标准

这一轮做完之后，项目应该达到下面状态：

- 正式 Web 已可发布
- 手机和平板浏览器可稳定使用
- 正式数据同步不再停留在“最小可用”
- PWA 和原生壳有明确落地前提
- 项目从“主线收口完成态”进入“跨端交付准备态”

## 一句话总结

接下来最正确的计划不是“继续堆功能”，而是：

`先把 Web 发布基础和移动端交付做实，再补完整同步，最后进入 iOS / Android 商店发布`
