# 页面功能清单

## 文档目的

这份文档提供给 UI 和 UX 设计师使用。

目标不是解释代码，而是把当前每个页面已经存在的：

- 页面目标
- 功能模块
- 交互逻辑
- 数据处理
- 状态反馈

拆成一份可以直接用于页面设计、交互设计和状态设计的输入材料。

## 产品主线

当前项目围绕一条单一主线展开：

`制定计划 -> 开始计划 -> 执行当天训练 -> 记录训练结果 -> 回到仪表盘查看历史与统计`

这意味着当前所有页面设计都应该围绕这条主线服务，而不是发散到社区、内容消费、推荐等无关方向。

## 产品约束

设计时必须默认以下前提成立：

- 单用户
- 单活跃计划
- 当前只有一个核心业务域：训练记录
- 训练草稿只保存在前端本地
- 正式训练日志、用户计划、计划 definition 已进入后端正式存储
- 正式训练结果只能通过后端原子接口提交

## 页面 1：训练仪表盘

### 页面定位

这是首页，也是产品总控台。

用户进入应用后，首先看到的是这里。
它承担四个职责：

- 展示当前活跃计划
- 引导进入今日训练
- 展示历史与训练统计
- 提供计划创建与计划浏览入口

### 主要模块

#### 1. 页面标题区

**功能**

- 展示页面标题
- 提供“创建新计划”入口

**交互逻辑**

- 点击“创建新计划”进入新建计划页

**数据处理**

- 无复杂数据依赖
- 只承担导航入口

#### 2. 活跃计划 Hero 区

**功能**

- 展示当前唯一活跃计划
- 展示总体进度
- 展示今日目标
- 提供“开始今日锻炼”入口

**交互逻辑**

- 如果存在活跃计划：
  - 显示计划名称
  - 显示计划描述
  - 显示当前已完成天数 / 总天数
  - 显示今日训练日名称
  - 点击“开始今日锻炼”进入训练会话页
- 如果不存在活跃计划：
  - 展示空状态
  - 提示用户去创建或选择计划

**数据处理**

- 数据来源：当前活跃用户计划 + 对应计划定义
- 页面需要展示派生数据：
  - 当前进度百分比
  - 当前训练日名称
  - 用户计划实例 ID

**设计关注点**

- 这是首页第一视觉焦点
- 需要明显区分“有活跃计划”和“无活跃计划”两种首屏
- CTA 优先级必须最高

#### 3. 训练统计区

**功能**

- 展示训练容量趋势
- 支持周/月切换

**交互逻辑**

- Tab 1：周容量
- Tab 2：月容量
- 鼠标悬停柱状图时显示数值提示

**数据处理**

- 数据来源：训练日志列表
- 数据加工：
  - 周维度聚合
  - 月维度聚合
- 当前统计重点是趋势，不是深度分析

**设计关注点**

- 统计区是“回顾区”，不是“行动区”
- 视觉上不应压过活跃计划区
- 空数据状态要明确

#### 4. 历史记录区

**功能**

- 展示历史计划
- 支持按状态筛选
- 提供恢复 / 重开 / 归档操作

**交互逻辑**

- 筛选项：
  - 全部
  - 已完成
  - 未完成
- 中断计划：
  - 可恢复继续
  - 可重新开始
  - 可删除记录
- 已完成计划：
  - 可再次挑战
  - 可删除记录

**数据处理**

- 数据来源：历史用户计划列表
- 数据加工：
  - 按状态过滤
  - 按时间排序
  - 显示完成状态或当前进度

**设计关注点**

- 必须明显区分 `completed` 和 `interrupted`
- 历史卡片的操作按钮要避免误触
- 恢复和重开不是同一个动作，视觉与文案要清楚分开

#### 5. 计划库区

**功能**

- 展示可用训练计划定义
- 提供进入计划详情页的入口

**交互逻辑**

- 点击卡片进入计划详情页

**数据处理**

- 数据来源：计划定义列表
- 显示字段：
  - 计划名称
  - 计划简介
  - 周期天数

**设计关注点**

- 这是“发现更多计划”的区域
- 卡片需要强调“可浏览、可点击”
- 空状态要有占位反馈

### 首页总体状态

UI/UX 设计时至少要覆盖这些状态：

- 有活跃计划
- 无活跃计划
- 有历史记录
- 无历史记录
- 有统计数据
- 无统计数据

## 页面 2：创建新计划

### 页面定位

这是一个结构化编辑页，不是普通单表单页。

用户在这里创建的是：

- 一个训练计划定义
- 并且提交后会直接启动为当前用户的新计划

### 主要模块

#### 1. 顶部导航区

**功能**

- 返回上一页
- 提供取消操作
- 提供提交操作

**交互逻辑**

- 点击返回或取消：离开页面
- 点击保存：提交当前计划

**数据处理**

- 保存前需要校验表单有效性

#### 2. 基本信息区

**功能**

- 输入计划名称
- 输入计划描述

**交互逻辑**

- 名称必填
- 描述选填

**数据处理**

- 输出字段：
  - `name`
  - `description`

#### 3. 训练日程区

**功能**

- 配置计划的结构化日程
- 支持两类节点：
  - 训练日
  - 循环

**交互逻辑**

- 可以新增训练日
- 可以新增循环
- 循环内部还可以继续新增项目
- 每个训练日可以继续添加动作
- 动作支持填写：
  - 名称
  - 组数
  - 次数

**数据处理**

- 最终输出的是 `schedule` 数组
- 结构是递归型，不是扁平列表

**设计关注点**

- 这是当前项目里最需要“结构感”的页面
- 设计必须清楚表达层级关系
- 用户必须能直观看出：
  - 这是一整天训练
  - 这是一段循环
  - 这个循环会重复几次

### 新建页总体状态

UI/UX 设计时至少要覆盖这些状态：

- 初始空表单
- 已填写但未保存
- 保存中
- 保存失败
- 表单校验失败

## 页面 3：计划详情

### 页面定位

这是计划浏览与启动决策页。

它不是编辑页。
它的作用是帮助用户理解一个计划，然后决定是否开始。

### 主要模块

#### 1. 顶部标题区

**功能**

- 返回上一页
- 显示计划名称
- 提供“开始此计划”入口

**交互逻辑**

- 点击返回：离开详情页
- 点击开始：为当前用户创建新的计划实例，并回到仪表盘

**数据处理**

- 数据来源：计划定义

#### 2. 计划简介区

**功能**

- 展示计划的文字描述

**交互逻辑**

- 无复杂交互

**数据处理**

- 显示 `description`

#### 3. 日程预览区

**功能**

- 只读展示整个训练日程
- 复用新建页的结构，只是不可编辑

**交互逻辑**

- 用户浏览周期结构、训练日和动作配置

**数据处理**

- 数据来源：计划定义
- 页面需要把计划结构转换为只读可展示形式

**设计关注点**

- 这是“理解计划”的区域
- 需要突出结构和内容，但不要让它看起来像编辑器
- “开始此计划”仍然应该是主要 CTA

### 详情页总体状态

UI/UX 设计时至少要覆盖这些状态：

- 加载中
- 找到计划
- 未找到计划 / 异常状态

## 页面 4：训练会话页

### 页面定位

这是当前产品最核心的执行页。

它承接的是真实训练过程，不是浏览，不是编辑，而是高频输入和过程反馈。

### 主要模块

#### 1. 顶部训练概览区

**功能**

- 显示当前计划名称
- 显示今天训练日名称
- 提供返回入口

**交互逻辑**

- 用户进入页面后先确认“今天练什么”

**数据处理**

- 数据来源：
  - `userPlanId`
  - 当前用户计划的 `currentDayIndex`
  - 对应计划定义展开后的训练日

#### 2. 动作卡片区

**功能**

- 按动作展示今天的训练内容
- 每个动作下展示多组输入项

**交互逻辑**

- 每一组都支持录入：
  - 次数
  - 重量
  - 是否完成
- 勾选完成后会触发休息计时器

**数据处理**

- 页面根据训练日配置动态生成表单
- 每个动作都带有：
  - 动作名称
  - 建议组数
  - 建议次数
  - 每组当前输入值

**设计关注点**

- 这是高频录入区
- 录入密度高，必须减少视觉干扰
- 重点是：
  - 快速填写
  - 快速确认完成
  - 快速理解每组状态

#### 3. 历史参考区

**功能**

- 在当前动作上下文中展示该动作上次训练表现

**交互逻辑**

- 用户不需要跳页查看
- 直接在当前动作卡片中看到上次记录

**数据处理**

- 数据来源：后端训练日志
- 查询粒度：按 `exerciseId` 读取最近一次训练记录
- 展示内容：
  - 上次训练日期
  - 每组重量与次数

**设计关注点**

- 这是辅助信息，不是主输入区
- 视觉层级要低于当前输入
- 但必须足够可读，因为它直接影响训练判断

#### 4. 草稿恢复与状态提示

**功能**

- 自动保存训练过程
- 重新进入页面时自动恢复
- 在界面中明确提示“当前是本地草稿，不是正式记录”

**交互逻辑**

- 用户录入后，表单会自动保存
- 再次进入相同训练日时，恢复上次未完成内容
- 如果训练日已经变化，则旧草稿失效
- 正式提交成功后才清理草稿

**数据处理**

- 草稿保存在本地
- 校验条件：
  - `userPlanId`
  - `trainingDayId`
  - `currentDayIndex`
- 草稿只属于过程态，不进入正式统计和正式历史

**设计关注点**

- 这是“无感恢复”能力
- 页面可以不强调保存按钮
- 但需要让用户知道：恢复的是草稿，不是已经提交成功的记录
- 草稿提示不应喧宾夺主，但必须清楚可见

#### 5. 底部固定操作栏

**功能**

- 跳过今日训练
- 完成训练

**交互逻辑**

- 点击“跳过今日训练”：
  - 二次确认
  - 通过原子接口提交 `skipped`
  - 成功后推进计划并返回仪表盘
- 点击“完成训练”：
  - 校验表单
  - 仅提交已勾选完成的组
  - 通过原子接口写入训练日志并推进计划
  - 成功后清理草稿
  - 返回仪表盘

**数据处理**

- 正式训练结果写入后端
- 正式训练结果和计划推进属于同一条原子链路
- 页面只保留本地草稿作为过程态

**设计关注点**

- 操作栏必须始终可达
- “完成训练”应该是主操作
- “跳过今日”应次一级，但不能隐藏

#### 6. 错误反馈区

**功能**

- 显示训练提交失败或跳过失败的错误信息

**交互逻辑**

- 当提交失败时，页面内展示错误，而不是只在控制台输出

**数据处理**

- 错误状态来自提交请求结果

**设计关注点**

- 错误要出现在用户看得见的位置
- 反馈应简洁、直接、可恢复

### 训练会话页总体状态

UI/UX 设计时至少要覆盖这些状态：

- 加载中
- 正常训练中
- 已恢复草稿
- 草稿提示可见
- 无历史参考
- 有历史参考
- 提交中
- 提交失败
- 跳过确认

## 全局组件：组间休息计时器

### 组件定位

这是一个跨页面浮层，不属于单页内容。

### 功能

- 勾选某组完成后自动启动 90 秒倒计时
- 显示剩余时间
- 支持：
  - 增加 30 秒
  - 直接结束

### 交互逻辑

- 默认不显示
- 触发后覆盖在页面上层
- 用户可以主动延长或提前关闭

### 数据处理

- 纯前端状态
- 不写后端

### 设计关注点

- 它是训练节奏辅助工具，不应喧宾夺主
- 但在训练场景下必须明显可见

## 给 UI/UX 的总设计优先级

### 优先级 1：训练主流程

- 首页活跃计划
- 开始今日训练
- 训练会话高频输入
- 完成训练反馈

### 优先级 2：计划创建与理解

- 创建新计划
- 理解计划结构
- 选择并开始计划

### 优先级 3：复盘与回看

- 历史计划
- 训练容量趋势
- 动作上次记录

## 设计时必须考虑的关键状态

- 空状态
- 加载状态
- 成功状态
- 失败状态
- 无数据状态
- 草稿恢复状态
- 提交中状态

## 一句话结论

当前这个产品不是“内容浏览型健身应用”，而是“训练执行型工具”。

所以 UI/UX 设计的核心应该围绕：

- 快速进入今日训练
- 低阻力完成记录
- 清晰获得进度反馈
- 顺手查看历史趋势

而不是把重点放到装饰性信息或复杂内容消费上。
