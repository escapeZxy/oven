# Roadmap

## 文档目的

这份文档不是画饼。

它只定义三件事：

- 现在已经完成到哪一层
- 接下来该先做什么
- 哪些事情现在不要碰

如果一个阶段不能直接提升主流程稳定性、数据可靠性或边界清晰度，就不要抢着做。

## 当前阶段判断

项目当前已经打通一条可用主线：

`计划管理 -> 当日训练 -> 本地草稿恢复 -> 原子提交正式训练结果 -> 历史查看 -> 基础统计`

这意味着项目已经过了“纯原型”和“后端骨架”阶段。

当前真正缺的不是“后端能不能存数据”，而是：

- 生产 Web 发布基础已经完成
- 多设备同步只完成了第一步
- 移动端交付壳还没开始
- 更深层训练分析还没开始

## Roadmap 原则

后续演进继续遵循四条规则：

1. 先保主流程，不做炫技
2. 先收紧边界，再扩功能
3. 先做正式数据可靠性，再做高级体验
4. 不同时启动多条主线

## 阶段 0：当前已完成

这一阶段已经具备：

- Angular 前端主流程
- `packages/core` 共享模型和业务规则
- 本地训练草稿恢复
- 基础训练容量统计
- NestJS + Prisma + SQLite 正式数据层
- `WorkoutPlan`、`UserWorkoutPlan`、`WorkoutLog` 正式持久化
- 仪表盘正式数据读取
- 训练会话原子提交
- 草稿与正式记录边界收口
- 真实认证主路径
- 项目文档体系

这一阶段的结果是：

- 你已经有一个真正能用的训练执行器
- 你已经有一条正式后端数据链路
- 你已经把认证接进正式链路
- 你已经有第一批服务端统计
- 但你还没有完整统计分析体系、完整多设备能力和正式跨端发布能力

## 阶段 1：收紧用户计划动作边界

当前状态：已完成

### 已完成结果

- `resume` 已提炼成显式动作
- `archive` 已提炼成显式动作
- 前端已切到动作型接口
- 生产路径中已不再依赖通用更新入口

## 阶段 2：引入真实用户体系

当前状态：已完成

### 目标

从 mock user 过渡到真实用户身份。

### 为什么排在现在

因为主流程的正式数据链路已经基本收口，认证终于不再只是给本地应用套壳。

### 本阶段应完成

- 注册 / 登录
- 用户会话管理
- 基础权限校验
- 用户数据隔离

### 技术判断

这一阶段可以选：

- 自建 NestJS auth
- Supabase Auth
- Firebase Auth

但无论选哪个，都不能破坏当前 `packages/core` 的模型边界和前端主流程。

### 已完成结果

- 登录 / 注册页已接入
- Bearer Token 鉴权已接入
- 会话恢复 / 注销已接入
- `user-plans` / `workout-logs` 已改为从认证上下文解析用户

## 阶段 3：完善服务端统计

当前状态：已完成

### 目标

把当前前端做的基础统计逐步迁移为后端可复用统计能力。

### 为什么放在这里

因为统计建立在稳定正式数据和真实用户之上。

现在正式记录已经稳定，但没有真实用户和更完整的服务端接口之前，没必要急着把所有统计搬过去。

### 本阶段应完成

- 周容量统计 API
- 月容量统计 API
- 计划完成率 API
- 动作趋势统计 API

### 当前已完成

- 周容量统计 API
- 月容量统计 API
- 计划完成率统计 API
- 动作趋势统计 API
- 仪表盘已切到服务端聚合结果

### 当前剩余

- 更深层训练分析

### 本阶段可以考虑补充

- 单动作 PR
- 训练频率
- 肌群覆盖
- 连续训练天数

## 阶段 4：生产 Web 发布基础

### 目标

让项目先成为一个可正式上线的 Web 产品。

### 为什么排在这里

因为如果连正式 Web 都还没落地，后面的 PWA 和原生壳都没有交付底座。

### 本阶段应完成

- 生产环境变量收口
- 前端部署方案
- 后端部署方案
- 数据库云端方案
- 域名、HTTPS、CORS、健康检查

### 当前状态

已完成

### 已完成结果

- 前端已部署到 `https://oven-two.vercel.app`
- 后端已部署到 `https://oven-production.up.railway.app`
- 数据库已部署到 Turso
- CORS 已收口到 `https://oven-two.vercel.app`
- 注册 / 登录 / 会话恢复已通过正式环境验证
- 创建计划 / 开始计划 / 完成训练 / 跳过训练已通过正式环境验证
- 同步 / 历史记录 / 统计已通过正式环境验证
- 平板浏览器访问已通过验证

## 阶段 5：移动端 Web 与最小 PWA

### 目标

让同一套主流程在手机、平板和桌面都能可用，并具备最小安装能力。

### 这一步为什么放在这里

因为你的最终目标是 `Web + iOS + Android`，而 PWA 是从 Web 进入移动端交付的最短路径。

### 本阶段应完成

- 关键页面移动端适配
- 安全区和底部操作栏适配
- PWA manifest
- 图标与安装壳
- service worker 或等价静态缓存壳

### 当前已完成

- Angular Web 主流程已可运行

### 当前剩余

- 移动端交互适配
- PWA 壳层

## 阶段 6：多设备同步与冲突处理

### 目标

让用户在不同设备之间共享正式数据。

### 这一步为什么晚做

因为同步不是“接口通了”这么简单。

它会强迫你定义：

- 时间戳策略
- 幂等策略
- 冲突合并规则
- 草稿与正式记录边界
- 离线后的重放逻辑

这些问题只有在前几层稳定后才值得认真处理。

### 本阶段应完成

- 正式记录多端拉取
- 同步状态提示
- 重复提交幂等保护
- 基础冲突规则

### 当前已完成

- 正式计划 definition 增量拉取
- 正式用户计划增量拉取
- 正式日志增量拉取接口
- 仪表盘同步状态提示
- 手动“立即同步”入口
- 训练提交的请求级幂等保护
- 提交冲突后的基础自动恢复
- 冲突类型化返回语义

### 当前剩余

- 复杂冲突合并
- 跨设备草稿策略
- 离线重放

## 阶段 7：原生壳与商店发布

### 目标

在不重写业务前端的前提下，把产品发布到 iOS / Android 商店。

### 本阶段应完成

- 选定原生壳路线，例如 Capacitor
- 生成 iOS / Android 工程
- 图标、启动图、包名和签名配置
- 真机验证
- 商店提审材料与发布流程

## 暂时不要做的事

### 1. 不要同时扩多个业务域

现在项目主线是训练执行。

不要在这个阶段同时铺开：

- 饮食
- 体重
- 社交
- 打卡社区

### 2. 不要过早抽象成“大平台架构”

现在最怕的是为了看起来高级，提前设计一堆：

- 微服务
- 事件总线
- 复杂同步引擎
- 通用 BI 平台

这些都属于典型的过度设计。

### 3. 不要推翻现有前端主流程

现在正确的做法是继续收紧边界，不是重写已经打通的主链路。

## 近期建议执行顺序

如果只看最近两周，建议按下面顺序做：

1. 先补移动端 Web 与最小 PWA
2. 再补完整同步阶段剩余能力
3. 再完成 iPhone Safari 与 Android Chrome 真机验证
4. 再进入原生壳与商店发布
5. 最后再补更深层训练分析

做到这里，这个项目就会从“主流程已收口”进入“产品化能力开始落地”的状态。

## 成功标准

后续每做一个阶段，都用这三个问题验收：

- 主流程是不是更稳了
- 数据是不是更可靠了
- 边界是不是更清楚了

如果这三个问题里有两个答不上来，这件事大概率不该现在做。

## 一句话总结

当前最正确的路线不是“继续接后端”，而是：

`在正式数据链路、真实认证、核心服务端统计和生产 Web 发布都已稳定的前提下，先补移动端适配和 PWA，再补完整同步，最后进入 iOS / Android 商店发布`
