# 当前产品地图

## 文档目的

这份文档只回答四个问题：

- 当前项目的主线是什么
- 用户现在真的能完成哪些场景
- 哪些能力已经落成正式数据链路
- 哪些东西还没做，不能假装存在

它不是需求文档，也不是技术方案文档。

## 当前产品定位

`oven` 当前是一个训练执行型工具，不是内容型健身平台。

它的最终交付目标已经明确：

- Web
- iOS
- Android

但当前真实形态仍然是一个以 Angular Web 为主的应用，移动端和商店发布还没有真正落地。

它围绕一条很明确的主线展开：

`创建计划 -> 开始计划 -> 进入当天训练 -> 本地恢复草稿 -> 原子提交正式训练结果 -> 回到仪表盘查看历史与统计`

当前已经不是“只有页面壳”的原型，也不是“纯本地 mock”的玩具。
它已经具备正式后端数据链路和真实认证，并启动了多设备同步第一步，但草稿仍然只保存在本地。

## 当前交付形态

当前真正可交付的形态只有一类：

- Web

当前还不具备：

- 正式生产部署方案
- 最小 PWA 安装壳
- iOS 原生壳
- Android 原生壳
- 应用商店分发配置

## 当前已实现的核心场景

### 1. 进入训练仪表盘

用户进入应用后会落到训练仪表盘。

当前仪表盘承担五个职责：

- 展示当前活跃计划
- 展示历史计划
- 展示计划库
- 展示周容量、月容量、计划完成率和动作趋势统计
- 展示正式数据同步状态并允许手动同步

当前这些正式数据与同步状态入口已经统一到后端：

- 计划库：`GET /workout-plans?updatedAfter=...&limit=...`
- 活跃计划 / 历史计划：`GET /user-plans?updatedAfter=...&limit=...`
- 周容量 / 月容量统计：`GET /workout-logs/volume`
- 正式数据同步游标：`GET /workout-plans` / `GET /user-plans`

### 2. 创建新训练计划

用户可以创建一个新的训练计划 definition。

当前支持：

- 填写计划名称
- 填写计划描述
- 配置结构化 `schedule`
- 提交后创建新的计划 definition
- 同时为当前用户创建新的活跃计划实例

### 3. 浏览计划详情并开始计划

用户可以打开计划详情页，查看：

- 计划名称
- 计划简介
- 日程预览

然后点击“开始此计划”，为当前用户创建新的执行实例。

### 4. 查看当前活跃计划

系统会从用户计划中识别唯一活跃计划，并在仪表盘展示：

- 当前计划名称
- 当前已完成进度
- 今天应该执行的训练日

当前仍然遵守单活跃计划假设。

### 5. 开始今日训练

用户点击“开始今日锻炼”后进入训练会话页。

系统会根据：

- `userPlanId`
- 当前计划的 `currentDayIndex`
- 计划 definition 展开后的训练日

推导出当天应该练什么动作。

### 6. 训练过程中录入数据

训练会话页支持按动作和组录入：

- `reps`
- `weight`
- `completed`

这是当前产品最核心的高频输入环节。

### 7. 训练中断后恢复草稿

训练页会自动保存草稿到本地，并在重新进入相同训练日时恢复。

当前草稿恢复会校验：

- `userPlanId`
- `trainingDayId`
- `dayIndex`

这意味着本地草稿只服务于“恢复输入过程”，不会误恢复到错误训练日。

### 8. 查看动作的上次训练记录

训练会话页会按动作读取最近一次正式训练记录，展示给用户作为参考。

当前目的很明确：

- 帮助用户做最基础的渐进超负荷判断
- 不试图在这个阶段做深度分析

### 9. 完成当天训练

用户完成训练后，系统会：

- 只采集勾选为 `completed` 的组
- 通过 `POST /user-plans/:id/logs` 原子提交正式结果
- 在同一条链路中写入训练日志并推进用户计划
- 同一提交意图会携带稳定的 `clientRequestId`，避免重复写入正式日志
- 如果服务端判定正式数据已变化，会返回明确的冲突类型和恢复动作
- 前端会根据冲突类型决定是“同步后继续当前训练日”还是“返回仪表盘确认最新状态”
- 只有后端提交成功后才清理草稿
- 成功后返回仪表盘

### 10. 跳过当天训练

用户可以手动跳过今天训练。

系统会：

- 通过同一条原子提交接口记录 `skipped`
- 推进计划索引
- 清理草稿
- 返回仪表盘

### 11. 查看历史计划

仪表盘会展示历史计划，并区分：

- `completed`
- `interrupted`

用户可以按状态筛选。

### 12. 恢复中断计划

中断计划可以被恢复为新的活跃计划。

恢复时，系统会中断原有活跃计划，保持单活跃计划假设不被破坏。

### 13. 重新开始已有计划

用户可以对已有计划执行“重新开始”。

这不是恢复旧实例，而是新建一个新的执行实例。

### 14. 归档历史计划

用户可以归档历史计划，使其不再出现在默认历史列表中。

### 15. 查看训练容量统计

当前已有四种统计视角：

- 周容量
- 月容量
- 计划完成率
- 动作趋势

当前这四类统计里：

- 周容量和月容量已经从“前端读取全量日志后自行聚合”切到“后端按时间桶聚合后返回结果”
- 计划完成率已经切到“后端按 `user-plans` 状态聚合后返回结果”
- 动作趋势已经切到“后端按正式完成日志聚合最近动作容量后返回结果”

当前统计重点是趋势回看，不是完整 BI 系统。

## 当前数据与交互假设

### 单会话假设

当前已经有真实用户体系，但浏览器里同一时刻仍然只服务一个当前登录用户。

这意味着：

- 已有注册
- 已有登录
- 已有注销
- 已有基础权限边界
- 当前仍然没有多账号并行或团队场景

### 单活跃计划假设

同一时刻只允许存在一个活跃计划。

这避免了“今天到底练哪个计划”的脏分支。

### 草稿与正式记录分离

当前已经明确：

- 草稿只保存在前端本地
- 草稿只服务于恢复输入过程
- 正式训练记录只认后端
- 正式训练进度只能通过原子提交接口推进
- 正式计划 definition / 用户计划支持基于 `updatedAt` 的增量拉取

这是当前系统最重要的一条产品边界。

## 当前未实现场景

### 1. 完整多设备同步

当前已经有：

- 正式计划 definition 的增量拉取
- 正式用户计划的增量拉取
- 仪表盘同步状态提示
- 手动“立即同步”入口
- 训练会话提交的请求级幂等保护
- 基于“服务端为准”的基础冲突恢复
- 冲突后的草稿冲突备份

当前还没有：

- 跨设备草稿同步
- 复杂冲突合并
- 离线重放

### 2. 完整动作库

当前动作更偏计划内录入，还没有完整动作库系统。

尚未覆盖：

- 标准动作库
- 动作分类
- 肌群标签
- 搜索与复用

### 3. 深度训练分析

当前只有基础容量图。

尚未覆盖：

- 单动作 PR
- 周频率
- 肌群分布
- 疲劳趋势

### 4. 正式跨端发布

最终目标虽然是 `Web + iOS + Android`，但当前还没有：

- 生产 Web 发布配置收口
- PWA manifest / service worker
- iPhone 和 Android 的安装壳
- 商店图标、启动图、签名和提审材料

这意味着当前项目还不能被准确称为“已具备跨端发布能力”。

## 当前前后端职责边界

### 前端当前负责

- 页面路由和流程组织
- 训练过程交互
- 本地状态与表单状态
- 本地训练草稿
- 基础统计展示

### 后端当前负责

- 健康检查
- Prisma + SQLite 正式数据层
- 用户账号与会话持久化
- JWT 鉴权与当前用户解析
- 训练计划 definition 持久化
- 用户计划正式持久化
- 训练日志正式持久化
- 原子提交训练结果
- 正式训练进度的边界校验
- `resume / archive` 显式动作接口
- 正式数据增量拉取接口
- 周容量 / 月容量服务端聚合
- 计划完成率服务端聚合
- 动作趋势服务端聚合

### 后端下一阶段应该负责

- 更细粒度的冲突合并规则
- 更完整的同步失败恢复策略

## 建议的产品演进顺序

如果继续沿着现有主线做，推荐顺序如下：

1. 先完成生产 Web 发布基础和移动端 Web 适配
2. 再补最小 PWA
3. 再补完整多设备同步
4. 最后进入 iOS / Android 原生壳与商店发布
5. 再补更深层训练分析

原因很简单：

- 先让用户真的能在各类设备上用起来
- 再补跨设备一致性
- 最后再进入更重的原生分发能力

## 一句话总结

当前项目已经打通了一条真实可用的训练执行主线：

`计划管理 -> 当日训练 -> 本地恢复草稿 -> 原子提交正式结果 -> 历史回看 -> 基础统计`

现在它最需要的不是“继续堆功能”，而是：

- 先把 Web 发布和移动端交付基础做实
- 再在稳定正式数据之上补完整同步和更深层分析
