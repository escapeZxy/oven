# 架构总览

## 文档目的

这份文档只回答四个问题：

- 当前项目分成哪几层
- 每一层负责什么
- 正式数据和本地过程态怎么分工
- 最终要怎样支撑 Web、iOS、Android 的统一交付

如果一个职责说不清放哪层，那说明边界还不够干净。

## 当前整体结构

项目当前分成三块：

- `src/`
  - Angular 前端应用
- `packages/core/`
  - 前后端共享的模型、契约和纯业务规则
- `server/`
  - NestJS 后端 + Prisma + SQLite

这是一个“前端主导交互、后端承接正式事实、core 承担领域规则”的 monorepo 结构。

但要明确一点：

最终的跨端交付目标不是再加两套前端，而是：

- 继续以 Angular 前端作为单一主交付面
- 先交付 Web
- 再通过 PWA 和原生壳把同一套主流程送到 iOS / Android

## 当前分层职责

## 1. Angular 前端

位置：

- `src/app`

前端现在不只是 UI，它负责：

- 路由组织
- 登录 / 注册与路由守卫
- 页面状态管理
- 用户交互
- 训练过程编排
- 页面数据读取与刷新
- 正式数据同步状态展示与手动触发
- 本地草稿存储
- 基础统计展示
- 后续 Web / PWA / 原生壳共用的前端主流程

### 前端内部结构

当前主要可以拆成四类：

- `pages/`
  - 页面级功能流入口
- `components/`
  - 纯展示或局部交互组件
- `services/`
  - 前端本地能力，如草稿与统计
- `data-access/`
  - 面向页面的 facade 层

### 关键判断

`WorkoutLogService` 是当前前端真正的应用服务入口。

它负责：

- 组织首页和会话页需要的 ViewModel
- 管理正式数据的后端读取
- 管理正式数据同步游标、合并与状态
- 包装训练会话原子提交
- 屏蔽页面对后端和 `core` 细节的直接感知

页面不直接碰仓储和后端细节，这一点是对的。

## 2. packages/core

位置：

- `packages/core/src`

这一层承担“与框架无关的业务定义”。

它当前主要包含：

- `models/`
  - 领域模型
- `repositories/`
  - 仓储契约
- `services/`
  - 领域服务
- `infrastructure/`
  - 当前仍保留一部分本地实现

### 它应该承担的职责

- 定义核心数据结构
- 定义仓储契约
- 承载训练计划展开、计划推进、日志结构等业务规则
- 尽量不依赖 Angular 和 NestJS

### 它当前已经做到的事

- 训练计划展开
- 当前训练日推导
- 用户计划状态模型
- 训练日志结构定义
- 历史和计划相关的纯规则

### 它当前仍然混着的东西

`infrastructure/` 里仍有偏前端的本地存储实现。

这意味着：

- `core` 还没有完全摆脱前端落地痕迹
- 但它已经具备明确的复用价值

这不是致命问题，但说明它还没到最终纯净形态。

## 3. NestJS 后端

位置：

- `server/src`

后端现在已经不再是骨架。

它当前负责：

- 服务启动和健康检查
- Prisma 客户端生命周期管理
- SQLite 正式数据存储
- `User` 账号与密码哈希持久化
- `AuthSession` 会话持久化
- JWT 签发与 Bearer Token 鉴权
- `WorkoutPlan` 持久化
- `UserWorkoutPlan` 持久化
- `WorkoutLog` 持久化
- 原子提交训练结果
- 正式训练进度边界校验
- 正式资源的 `updatedAt` 增量拉取
- 提交冲突时返回服务端为准的正式状态
- 提交冲突时返回明确的 `code + recoveryAction`
- 周容量 / 月容量服务端聚合
- 计划完成率服务端聚合
- 动作趋势服务端聚合

它当前还没有负责：

- 跨设备草稿同步
- 复杂冲突合并
- 更完整的同步失败恢复策略
- 生产环境发布能力
- PWA 壳层能力
- iOS / Android 原生壳能力

## 当前数据流

## 1. 页面到 facade

当前默认主链路是：

- 页面组件
- `WorkoutLogService`
- HTTP API
- NestJS Service
- Prisma
- SQLite

这是现在最合理的一层隔离。

## 2. 草稿与正式记录分流

当前系统最关键的架构事实不是“是否前后端分离”，而是：

- 本地草稿和正式记录已经明确分流

具体是：

- 训练草稿只保存在前端 `WorkoutSessionDraftService`
- 草稿绑定 `userPlanId + trainingDayId + dayIndex`
- 草稿只用于恢复当前训练输入过程
- 正式训练记录只能通过 `POST /user-plans/:id/logs` 写入后端

这条边界已经收口，是当前系统能稳定演进的前提。

## 3. 正式提交链路

当前正式训练结果的提交链路是：

- 页面收集用户输入
- 前端只提取勾选为 `completed` 的组
- `WorkoutLogService.submitWorkoutSession()`
- 前端为同一提交意图生成稳定的 `clientRequestId`
- Bearer Token 提供用户身份
- `POST /user-plans/:id/logs`
- 后端事务内写入 `WorkoutLog`
- 后端事务内推进 `UserWorkoutPlan.currentDayIndex/status/isActive`
- 冲突时前端按 `recoveryAction` 强制同步或返回仪表盘
- 冲突前的本地输入会转存为草稿冲突备份，避免被新训练日覆盖

也就是说，正式记录和正式进度现在属于同一条原子链路。

## 4. 正式边界校验

后端现在已经开始承担“防止脏状态进入正式数据层”的职责。

当前校验包括：

- `Authorization: Bearer <token>` 必须可解析为有效用户
- `user-plans` / `workout-logs` 从身份上下文取用户，不再信任客户端传入 `userId`
- 提交的 `trainingDayId` 必须匹配服务端当前计划进度
- 每次提交必须只推进一天
- `completed` 至少要有一组完成数据
- `skipped` 不能携带 `completedExercises`
- 相同 `clientRequestId` 的重复提交只能落库一次
- 不能通过 `POST /workout-logs` 绕过原子提交链路
- 恢复和归档通过显式动作接口处理，而不是通用 `PATCH`

这说明后端现在不只是存数据，已经开始保护边界。

## 5. 统计数据流

当前统计已经拆成“后端聚合 + 前端展示适配”两层：

- 前端调用 `GET /workout-logs/volume`
- 后端按周或月时间桶聚合正式日志
- 前端只把聚合结果适配为图表数据
- 图表组件继续只负责渲染

这一步是对的，因为：

- 正式统计口径开始统一
- 前端不再为了图表持有全量日志聚合逻辑
- 图表组件本身不用跟着重写

但这还不是终点。

目前已经迁过去的只有：

- 周容量
- 月容量
- 计划完成率
- 动作趋势

后续仍要继续补：

- 更深层训练分析

## 6. 正式数据同步第一步

当前已经不是“刷新页面全量重读”的粗暴模型。

现在正式数据同步的最小链路是：

- 前端保存 `workout-plans` 与 `user-plans` 的同步游标
- 仪表盘刷新或用户点击“立即同步”时发起增量请求
- `GET /workout-plans?updatedAfter=...&limit=...`
- `GET /user-plans?updatedAfter=...&limit=...`
- 前端按 `id` 做 upsert 合并
- 游标推进到本批返回结果中的最大 `updatedAt`

这一步只解决一件事：

- 不同设备之间拉到同一份正式数据

它明确不解决：

- 草稿跨设备同步
- 冲突合并
- 离线重放

## 7. 跨端交付目标

当前代码结构虽然只有三块，但最终交付形态会有四层含义：

- Angular 应用
  - 单一业务前端
- Web / PWA 交付层
  - 浏览器访问和桌面安装
- 原生壳交付层
  - iOS / Android 商店壳
- NestJS + 数据层
  - 统一正式事实源

这里最重要的判断是：

- Web、PWA、iOS、Android 不应该各自拥有一套业务事实
- 原生壳只是交付壳，不应该成为第二套业务前端

## 当前边界判断

## 适合留在前端的

- 页面状态
- 表单状态
- 训练会话高频输入
- 本地草稿恢复
- 当前 UI 层统计展示
- 路由切换与页面编排

## 适合留在 core 的

- 领域模型
- 计划展开规则
- 日志结构
- 计划状态切换规则
- 仓储契约定义

## 应由后端负责的

- 正式数据持久化
- 正式训练进度推进
- 正式训练记录写入
- 用户身份与权限
- 正式数据增量同步接口
- 服务端统计接口

## 当前最大的架构事实

当前项目最真实的状态不是“三层都成熟”，而是：

- 前端成熟度最高
- `core` 已经具备复用价值
- 后端已经承接正式事实源
- 主流程最大的边界已经从“接后端”推进到“同步与正式事实的一致性”

所以现在最不该做的事是：

- 在后端重复抄一套领域规则
- 绕开 `core` 重造模型
- 重新让正式记录走双写链路
- 在没有明确同步策略前同时保留多条正式事实源

## 推荐演进方向

正确演进不是“把所有东西都搬到后端”，而是继续把职责收拢到正确的位置。

建议顺序如下：

1. 保持页面继续只调用前端 facade
2. 保持正式训练结果只走原子接口
3. 保持 `user-plans` 继续只暴露显式动作接口
4. 先完成生产 Web 发布基础和移动端 Web 适配
5. 再补最小 PWA
6. 在已落地的正式数据增量拉取之上补复杂冲突合并和恢复策略
7. 最后再进入 iOS / Android 原生壳与商店发布
8. 再补更深层分析

## 理想终态

最终应该形成下面这个结构：

- Angular
  - 负责页面、交互、本地过程态和单一业务前端
- Web / PWA / 原生壳
  - 负责不同终端的交付外壳
- `packages/core`
  - 只负责模型、契约、纯业务规则
- NestJS
  - 只负责认证、正式持久化、同步接口、统计和动作型 API

这才是一个干净的全栈结构。

## 一句话总结

当前项目的正确描述是：

`前端主导交互 + core 承担领域规则 + 后端承接正式事实源`

现在最大的任务已经不是“让后端能存数据”或“把认证接上”，而是把发布基础、同步能力和跨端交付壳收拢到对的地方。
