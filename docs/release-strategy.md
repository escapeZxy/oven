# 发布与跨端路线

## 文档目的

这份文档只回答四个问题：

- 项目的最终交付目标是什么
- 当前距离正式发布还差什么
- Web、PWA、iOS、Android 应该按什么顺序推进
- 哪些路线现在不要碰

它不是部署脚本，也不是商店提审清单。
它定义的是一条切合当前代码现实的发布路线。

如果要直接执行当前阶段，请配合阅读：

- `docs/production-web-release-runbook.md`

## 最终目标

`oven` 的最终目标不是只做一个浏览器里的训练工具。

最终要形成的是同一套核心产品能力在三类交付形态上的统一交付：

- Web
- iOS
- Android

这里的“统一交付”不是三套前端各写一遍。

正确路线是：

- 共享同一套业务模型和正式事实源
- 共享同一套后端 API 和认证体系
- 尽量共享同一套 Angular 前端主流程
- 通过不同交付壳层进入浏览器、PWA 和应用商店

## 当前真实状态

当前项目已经具备：

- Angular 前端主流程
- NestJS 后端正式数据层
- 真实认证
- 服务端统计
- 正式数据最小同步链路
- 训练提交幂等保护
- 基础冲突恢复与冲突分类语义

但当前还不具备正式发布基础：

- 还没有生产环境 API 配置收口
- 还没有正式部署方案文档化
- 还没有 HTTPS / 域名 / CORS 收口
- 还没有 PWA manifest / service worker / 安装壳
- 还没有 iOS / Android 原生壳
- 还没有商店图标、启动图、签名和提审配置

一句话说清：

`现在已经具备产品主流程，但还没有具备正式跨端发布能力`

## 当前交付判断

基于当前代码状态，正确的交付判断只有一个：

- 先把 Web 正式上线
- 再补最小 PWA
- 最后再接 iOS / Android 原生壳

反过来做基本都会翻车。

原因很简单：

- 当前代码本质上仍然是一个 Angular Web 应用
- PWA 和原生壳都建立在稳定 Web 主流程之上
- 如果现在直接冲商店，会把发布问题、移动端体验问题和同步问题搅成一锅粥

## 跨端交付原则

后续发布继续遵循五条规则：

1. Web 是第一交付面，不先跳过 Web 去做商店壳
2. 同一套正式事实源服务 Web / PWA / iOS / Android
3. 不为上架单独重写一套前端
4. 不在发布阶段引入新的双写或双事实源
5. 先保证用户可用，再追求“像原生 App 一样”

## 分阶段发布路线

## 阶段 A：生产 Web 发布基础

目标：

让项目先以正式 Web 形态可发布、可访问、可维护。

必须完成：

- 前端 production 环境变量收口
- 后端 production 环境变量收口
- 云端数据库方案明确
- 前端静态托管方案明确
- 后端服务托管方案明确
- 域名、HTTPS、CORS、健康检查可用
- 基础日志与错误排查入口可用

交付结果：

- iPhone / Android / 平板浏览器可以直接访问
- Web 成为唯一正式交付面

## 阶段 B：移动端 Web 适配

目标：

让同一套 Web 在手机和平板上真正可用，而不是“能打开”。

必须完成：

- 关键页面响应式收口
- 训练页小屏输入体验优化
- 安全区和底部操作栏适配
- 横竖屏与平板布局校验
- 基础触控反馈和可点击区域优化

交付结果：

- Web 在手机和平板浏览器上可稳定使用
- 不依赖原生壳也能交付真实用户

## 阶段 C：最小 PWA

目标：

在稳定 Web 的基础上提供“可安装”的类 App 体验。

必须完成：

- `manifest.webmanifest`
- 应用图标与 `theme-color`
- Angular service worker 或等价静态资源缓存方案
- 基础离线壳层可打开
- 安装入口和图标表现验证

明确边界：

- 不把 PWA 等同于完整离线能力
- 不在这一阶段引入离线提交重放
- 不在这一阶段修改正式事实源

交付结果：

- Android 可安装到主屏
- iOS Safari 可添加到主屏
- Web 仍然是主交付面，PWA 是增强壳层

## 阶段 D：完整多设备同步

目标：

让 Web / PWA / 后续原生壳看到同一份正式数据。

必须完成：

- 复杂冲突合并规则
- 更完整的同步失败恢复
- 草稿跨设备策略做出明确取舍
- 是否支持离线重放做出明确取舍

注意：

- 如果这一阶段没做完，PWA 和原生壳也只是“多入口”，不是完整多端体验

## 阶段 E：原生壳与商店发布

目标：

在不重写业务前端的前提下，把现有产品送进 iOS / Android 商店。

建议路线：

- 基于 PWA 稳定主流程后接 Capacitor
- 生成 `ios/` 与 `android/` 工程
- 补原生图标、启动图、包名、签名、提审配置
- 做商店审核所需的隐私、账号、截图、说明材料

必须完成：

- 原生壳集成验证
- 登录态、路由、深链和网络异常校验
- iOS / Android 真机验证
- TestFlight / Play 内测流程跑通

交付结果：

- Web 继续可用
- iOS / Android 获得原生安装和商店分发能力

## 当前最合理的执行顺序

如果从今天开始按最终目标推进，最合理顺序是：

1. 先完成生产 Web 发布基础
2. 再补移动端 Web 适配
3. 再补最小 PWA
4. 再补完整多设备同步
5. 最后接原生壳和商店发布

## 现在不要做的事

下面这些路线现在不要碰：

- Flutter 重写
- React Native 重写
- 同时维护 Web 和原生双前端
- 为了移动端提前拆微服务
- 在没定义清楚同步边界前硬做完整离线模式

原因只有一个：

这些都不能更快把产品送到用户手上，反而会把复杂度炸穿。

## 成功标准

最终目标不是“代码看起来像跨端项目”，而是下面四件事同时成立：

- 同一个账号在 Web、iOS、Android 都能用
- 正式数据口径一致
- 主流程行为一致
- 发布和更新路径可维护

## 一句话总结

`oven` 的跨端路线应该是：先把 Web 发布做实，再补 PWA，再在稳定同步和稳定主流程之上进入 iOS / Android 商店分发。`
